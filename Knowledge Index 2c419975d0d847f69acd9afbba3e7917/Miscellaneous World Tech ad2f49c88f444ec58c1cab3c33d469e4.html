Looping sounds with an intro section

If you need to play a looping sound with an intro section, you can create a single audio .wav file that contains the intro followed immediately by the loop, and use UniWaveLoop to define where the loop starts.

My world VRWare uses this.

See: https://github.com/kraiHD/UniWaveLoop

List all of your worlds on VRChat website

Don't go to the World page; go to your Profile page, i.e. https://vrchat.com/home/user/usr_505d0888-f9f1-4ba1-92d5-71ff09607837

Then, expand the Worlds section at the bottom. Unlike the Worlds page, this does not require scrolling the list sideways.

Don't build too far away from the Origin of the world

When building too far away from the Origin of the world, precision issues will cause meshes to deform or move noticeably different. This is especially visible in VR where precision matters.

The main stage where players are expected to hang out most of the time should be close to the origin of the world, where the (0, 0, 0) point is.

Interacts and Pickups having an infinite range is caused by a trigger collider located between them

Interacts and Pickups may sometimes have an infinite range, if there is a trigger Collider located anywhere between the player's hand and that Interact or Pickup.

Disable the Collider of a Canvas to disable raycasting from an infinite range

UI Canvas like menus can be raycast with an infinite range, and will show the laser. You can disable the Collider component of a Canvas with Udon to disable the raycast and the laser.

If you disable the Graphics Raycaster component, clicking won't produce events, but the raycast laser will still show up. This is probably not what you want.

Prevent camera from clipping player's faces when two players' faces are very close to each other

If your world is a small chill world, set the Main Camera's Near Clipping Plane to the minimum possible value. This will prevent two player's faces from clipping when they are very close to each other.

This advice may not apply if the world has extremely large open spaces.

Prevent scrolling UI when moving around in the world

Some scrolling UI may be accidentally scrolled when moving around in the world using the WASD navigation keys or the thumbstick.

To prevent this:

  • Navigation Mode on UI Elements should be set to None.
  • ScrollRect components should have a Scroll Sensitivity set to 0.

Main Camera settings may cause black reflections on mirror-like surfaces and incorrect Reflection Probes being sampled across rooms

If reflections on mirror-like surfaces are black, or if unexpected Reflection Probes are being sampled across rooms, check the Main Camera Rendering Path and make sure it is set to Use Graphics Settings.

This may happen when re-using a scene from an existing asset pack.

Maximum number of players

In the VRChat website, you can set a number of players without reuploading the world.

VRChat calls it the Capacity on the website.

However, this is not the maximum number of players that your world can hold.

The maximum is the following, summed together:

  • The Capacity (16 in the picture above). In public instances, the world will no longer show up in the public world instances list if the current player count of that world is equal or above the Capacity.
  • + The Capacity, again. Worlds will have double the capacity in normal conditions. Public instances can still be joined above that Capacity even if they are no longer displayed in the public world instances list.
  • + 1 reserved slot for the creator of the Instance. The creator of the Instance is the person that clicks Drop Portal or clicks New Instance, in-game or on the website. The Creator of the Instance can always join, even if the world is already at double the capacity.
  • + 1 reserved slot for the Author of the World. The author of the world is the person who uploaded the world. The Author of the World can always join, even if the world is already at double the capacity.

If the Creator of the Instance and/or the Author of the World are already in the instance, other regular players cannot join even if the world is exactly at double the Capacity.

The Maximum Capacity that can be set on VRChat website is 40. That means the absolute maximum player count is 82 in normal conditions.

A note about VRChat Team members

VRChat Team members are suspected to have reserved slots of their own, supposedly being able to break the maximum player count of a world.

However, this is not documented anywhere. If this is true, the onus in on VRChat Team members if they break a Udon world by joining it above maximum player count (i.e. player pools, player arrays, etc.).

You should not have to care about this as a world creator.

The maximum number of players ever publicly witnessed in a VRChat world is around 130.

Dynamic Bones are not simulated at long distances away from the player

The Dynamic Bones on avatars are forcefully not simulated when the player is at a long distance away from that other player. This also applies to the Dynamic Bones present in the world.

Networked IK fidelity of other players depends on their distance away from the player

The IK fidelity of other players depends on their distance away from the player. This will cause some issues when you are trying to leverage on the bone data at long distances, or if you are using cameras to capture players and play back those cameras on a render texture.

Prevent users from dropping portals

If you need to prevent users from dropping portals, there are two known ways:

  • Use Udon to move a Respawn point to be in front of the player. This will cause a regular Portal too close to spawn message to pop up. The JSR Festival world uses this.
  • Detect the portal and manipulate it with Udon. You can move it somewhere else. The process is detailed in the code below, it revolves around detecting a specific GameObject name in the Water layer. Destroying the portal may cause the system menu not to close after opening the portal, so I do not recommend doing it. The VRWare world uses this.
  • Show code
    using UdonSharp;
using UnityEngine;
using VRC.SDKBase;

[UdonBehaviourSyncMode(BehaviourSyncMode.None)]
public class RemoveSpawnedPortals : UdonSharpBehaviour
{
    private Vector3 _halfExtents;
    private Collider[] _buffer;
    private int _waterLayerMask;

    private void Start()
    {
        _halfExtents = Vector3.one * 15;
        _buffer = new Collider[32];
        _waterLayerMask = 1 << 4; // 4 is the Water layer, where portals are spawned
    }

    private void Update()
    {
        // This technique was documented several times on Prefabs Discord:
        // Detect objects called "PortalInternalDynamic" in the Water layer using the Physics.OverlapBox function.
        var size = Physics.OverlapBoxNonAlloc(transform.position, _halfExtents, _buffer, Quaternion.identity, _waterLayerMask);
        for (var i = 0; i < size; i++)
        {
            var that = _buffer[i];
            var itIsValid = Utilities.IsValid(that)
                          && Utilities.IsValid(that.gameObject);
            if (itIsValid)
            {
                var thatIsAPortal = that.gameObject.name.Contains("PortalInternalDynamic");
                if (thatIsAPortal)
                {
                    // Don't destroy the portal, move it somewhere else
                    that.gameObject.transform.position = that.gameObject.transform.position + Vector3.up * 20;

                    // For some unknown reason I have not investigated, the behaviour crashes in OverlapBoxNonAlloc on next update, so recreate the buffer.
                    _buffer = new Collider[32];

                    // Only stop at the first portal detected on this tick, which should be reasonable
                    // Spawning 2 portals on the same tick might never occur, and even if it did, it would be handled next tick
                    return;
                }
            }
        }
    }
}
