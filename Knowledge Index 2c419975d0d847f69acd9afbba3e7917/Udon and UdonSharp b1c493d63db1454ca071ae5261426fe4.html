UdonSharp - Make a Field editable but not accessible

If a field needs to be editable on the Inspector but not accessed by other UdonBehaviours, declare it [SerializeField] private

[SerializeField] private Vector3 YourField;

Internally, this is still considered to be a public variable, so it can be modified with editor scripts and Get/Set public variable. You won't be able to access it via a field access from other UdonSharp scripts, which is the point.

UdonSharp - Make a Field accessible but not editable

If a field needs to be accessed by other UdonBehaviours but not editable on the inspector, declare it [NonSerialized] public

[NonSerialized] public Vector3 YourField;

This is different to [HideInInspector] in that it may not have an unexpected value if its value was ever serialized into the UdonBehaviour. This will not show up in the Inspector when debugging in Play mode.

UdonSharp - Be more productive by editing and recompiling UdonSharp without refreshing the entire project

If you disable auto-refresh on the project, UdonSharp can still manage to recompile modified UdonSharp files automatically without requiring the entire project to be refreshed.

In order to refresh the project manually, press CTRL-R. You will need to do this if you add files to your project, or edit existing assets such as textures. You don't need to do this if you only edited the contents of a UdonSharp script; the Udon program will be recompiled.

In "Editor > Preferences > General":

  • Uncheck "Auto-Refresh".
  • In "Script Changes While Playing", switch to "Recompile After Finished Playing".

This modification will affect all of your projects, not just this one.

A small caveat is that some newly added Attributes will not be visible by Editor Scripts.

Use SyncMode.None whenever possible

If there is a very high number (think 500) of UdonBehaviours with SyncMode set anything other than None, the owner of the objects may not be able to send manual synchronizations for several seconds after a player joins.

Setting SyncMode.None could reduce load times because these objects are no longer networked.

You can set SyncMode.None if all the following are true:

  • Ownership is not used on that behaviour (SetOwner/GetOwner), including by other behaviours.
  • There are no Custom network events.
  • There are no Synced Variables.

Execution Order is not respected on nested UdonBehaviours

Execution order is not respected on UdonBehaviours that are nested with other UdonBehaviours in the hierarchy.

The fixed timestep in FixedUpdate() depends on the HMD's refresh rate

The fixed timestep in FixedUpdate() is not a constant across players.

It depends on the HMD's refresh rate.

Make sure you take the actual time step into account, and not assume a specific value.

This also means the physics simulation of Rigidbody objects is likely going to be different for all players.

Read a RenderTexture of a Camera

To read a RenderTexture of a Camera, attach a UdonBehaviour on the same object that holds the Camera component, and implement OnPostRender()

public class CameraReader : UdonSharpBehaviour
{
    [SerializeField] private Texture2D buffer;
    [SerializeField] private Rect rect;
    [PublicAPI] public float[] Output;

    private void Start()
    {
        Output = new float[5];
    }

    private void OnPostRender()
    {
        // The first pixel is the bottom left
        buffer.ReadPixels(rect, 0, 0, false);
        buffer.Apply(false);

        var pixels = buffer.GetPixels();
        Output[0] = pixels[0].r;
        Output[1] = pixels[1].r;
        Output[2] = pixels[2].r;
        Output[3] = pixels[3].r;
        Output[4] = pixels[4].r;
    }
}

The first pixel returned by GetPixels() is at the bottom left.

You may need to generate the Texture2D asset on your own using an editor script. General textures assets from files will not work; they cannot be written into.

Calling Interact on another UdonBehaviour

If for any reason you need to call the Interact() function of a UdonBehaviour from another UdonBehaviour (i.e. call Lura's Switch buttons without modifying Lura's Switch source code), manually call the _interact Custom Event on that behaviour:

behaviour.SendCustomEvent("_interact")

The same may be true for other events, a underscore _ followed by a lowercase letter of the event.

Use AlignRoomWithSpawnPoint to teleport the player without making them turn

When the player is looking up or down with their head tilted, it is likely that the cardinal direction of their body will not match the cardinal direction of their head.

This can cause some issues when teleporting. If done every frame, this can cause the player to spin out of control.

This can be alleviated by teleporting using the player's tracking origin (play space) as a reference instead.

void _TeleportTo(Vector3 destination)
{
    var origin = Networking.LocalPlayer.GetTrackingData(VRCPlayerApi.TrackingDataType.Origin);
    var relativePos = origin.position - Networking.LocalPlayer.GetPosition();
    Networking.LocalPlayer.TeleportTo(destination + relativePos, origin.rotation, VRC_SceneDescriptor.SpawnOrientation.AlignRoomWithSpawnPoint);
}

Networking.IsInstanceOwner has no effect

Networking.IsInstanceOwner was meant to return true if the player is the person who has created the instance (and they may not be present in the instance). This is different from the notion of instance master.

However, this currently has no effect. You must not use it.

See: https://feedback.vrchat.com/bug-reports/p/networkingisinstanceowner-true-for-all-users

Attach data to a Player using Player Object Pools

In order to let the player have additional networked information (i.e. Player Score, Player Health), and emit events (i.e. The player has killed another player), you need to sort-of create a behavior that represents the player.

Let's say you want to support 20 players. There are two popular ways to achieve this:

  • The first option is to create 20 objects. The player must click a button to participate. The player will gain ownership of exactly one of the objects. They can only have one object activated at any time, and will be able to use this object to broadcast their state. Remember that all networked objects must have an Owner, so a player will necessarily own multiple of these objects, so treat Ownership and the activated object as different notions.
  • The second option is to use an automatically managed pool of 20 objects. The 20 objects will be automatically be assigned to players as they join and leave the world. This is challenging to create, so you should probably not create your own. Use an existing implementation. Nowadays, I will recommend CyanPlayerObjectPool by CyanLaser which is a more robust solution to the popular SimplePlayerObjectPool by Phasedragon which I will no longer recommend but served well. See: https://github.com/CyanLaser/CyanPlayerObjectPool/

UdonSharp - Invoking a method on a UdonBehaviour that crashed will return the last returned object, which can hide the real error and mislead you

If a UdonBehaviour has crashed, calling a method that returns an object on that UdonBehaviour could return the last returned object. This can deviously create additional errors that are just the result of cascading effect.

For example, if a method returns a list of current players in the instance, calling that method again while it is crashed will return the same list, which may contain players that have left the instance.

In this case, the correct fix would have been to identify and resolve the cause of the crash; not to handle invalid players.

Play an event almost simultaneously for all players by syncing a Networking.GetServerTimeInMilliseconds() in the future

If you get the value for Networking.GetServerTimeInMilliseconds() + some delay in the future, and sync it, you can use it to trigger an event almost simultaneously for all players by orders of 10s of milliseconds, even with players across the world.

My world VRWare uses this.

InputJump will not fire when the Quick Menu is open

InputJump will not fire when the Quick Menu is open.

This is intentional.

See: https://vrchat.canny.io/vrchat-udon-closed-alpha-bugs/p/1134-inputjump-does-not-fire-while-quick-menu-is-open

OnPlayerLeft player is invalid, but can still access some information with it

The player provided in OnPlayerLeft is invalid, it will not pass a Utilities.IsValid call.

However, some information can still be derived from it.

  • You can get the leaving player name.
  • You can detect if the Local Player is leaving by checking if it is equal to Networking.LocalPlayer which can be used for playing a sound, or some more curious things like resetting the world skybox to the original material so that the VRChat loading screen doesn't become pink.
