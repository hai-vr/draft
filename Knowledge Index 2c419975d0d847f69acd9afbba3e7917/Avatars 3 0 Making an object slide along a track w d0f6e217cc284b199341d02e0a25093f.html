Avatars 3.0: Making an object slide along a track with Constraint Track Animations

This page is a work in progress, and is incomplete.

Warning, this is really hard

Constraint Track Animations is an advanced esoteric topic, and requires you to be comfortable with the following subjects:

  • Avatars 3.0
  • Animations
  • Constraints
  • Modeling
  • Vertex groups

The following subjects are recommended:

  • Programming in C# and Editor Scripting
  • Animated Animator Parameters

Creating Constraint Tracks require a lot of patience and doing a lot of mistakes. I cannot recommend creating constraint tracks unless you're obsessed about the subject.

I don't do most of this by hand

This page describes the fundamentals of how it works. However, most of what I describe is something that I've streamlined by automating most of their creation. This automation is a result of learning from my past mistakes, because doing this manually is painful and I learned it the hard way.

I really, cannot recommend anyone doing this by hand. Maybe, just once on a simple track to get the basics working.

Constraint Tracks moves a single object/bone along a track

A constraint track is a Parent Constraint that allows a bone to move along a track.

The Parent Constraint, once setup, looks typically like this:

This is an animated GIF.

This Parent constraint has weights. These weights can be animated by an animation.

The weights looks like this in the animation:

This animation describes that only two constraints are active at a time, and the sum of the weights of all constraints is always equal to 1.

The resulting effect is that the parent constraint is being linearly interpolated between each transform of the track. Since this is a parent constraint, the position and rotation are linearly interpolated.

The track is made of objects parented to other bones, and not necessarily the same ones

A main attraction of constraint tracks is the ability to re-parent a bone from a piece of apparel to a neighboring bone that was not designed for it.

For instance, the Wrist part of a piece of apparel can move along the lower arm instead of the hand wrist.

That same wrist part can then move along the upper arm instead of the hand wrist.

Therefore, you can make the Wrist part move from the hand wrist, to the lower arm, to the upper arm. The obtained effect here is pulling the sleeves.

This is different from blendshapes, because blendshapes will not help get you get rid of the movement influence from the original bone; the wrist in this case.

Creating a constraint track is tedious if not created through automation

Creating a constraint track is tedious, because you may need to do numerous adjustments in the path, the timing, the animation, etc.

Therefore it is not advisable to create a constraint track without the help of an automation tool.

In addition, this is a gimmick that requires a sizable amount of constraints. Constraints are fast, but after a critical number of active constraints are present in the scene, there will be a significant CPU performance dip.

For this reason, it is important to disable unused constraints. A constraint track only has 2 track points active at one time, so if the track points themselves uses constraints, the other unused track points can be disabled. Disabling constraints adds another painful step to the animation process.

The apparel model needs to be modified to detach from the standard Humanoid

In order to make apparel model use Constraint Tracks, they need to be modified so that the standard Humanoid bones are no longer used by the apparel.

Rather than having a separate armature, we remap the apparel bones to be children of the respective bones of the Humanoid armature. This is done so that constraints are not needed if the gimmick has never run. It also allows to overcome some execution order issues related to constraints.

This is done the following way, using Blender:

  • Let's take the "Hand" for example, for a Hoodie piece of apparel.
  • In the Hoodie mesh, the vertex group "Hand" needs to be changed to something else, like "Z_Hand_Hoodie." The "Z_" in the name here is to prevent a possible drifting issue in IK, and also to influence the bone into being exported as the last bone of the hierarchy. Having these non-Humanoid bones as the first bone of the hierarchy can cause issues.
  • Edit the Armature.
  • Duplicate the "Hand" bone. By default, the new name of the bone will be "Hand.001"
  • Rename the new bone "Hand.001" to be "Z_Hand_Hoodie"
  • Set the parent of the "Z_Hand_Hoodie" to be the "Hand" bone.

Use a neutral object to represent the initial position of the object being animated

If you're animating a piece on the apparel that isn't at the center of the bone itself, such as a zipper of a Hoodie, then the first constraint of the path (initially weighted at 1) should be a neutral transform representing the center of that thing (the zipper in this case).

This will allow you to define the points of the track in a more sensible way.

The apparel bones follow another bone

Don't create the Constraint Track directly on the apparel bones; instead, use a Parent Constraint on the apparel bone, and assign the source to be the Constraint Track.

The Constraint Track can be at an offset, and the offset is often the neutral object described earlier.
